## Context

**当前状态：**
- Go 微服务项目，基于 go-zero v1.4.4 框架构建
- 8 个 Go 模块：ucenter、market、exchange、jobcenter 及其 API 服务，加上 grpc-common、mscoin-common 公共模块
- 使用 go.work 工作区管理多模块依赖
- gRPC 用于服务间通信，Etcd 作为服务发现
- MySQL 8.0、Redis 6.2、MongoDB 4.2、Kafka 作为基础设施

**依赖问题清单：**

| 组件 | 当前版本 | 目标版本 | 严重性 | 理由 |
|------|----------|----------|--------|------|
| golang.org/x/crypto | v0.7.0 | v0.49.0+ | 🔴 紧急 | 多个已知 CVE 漏洞 |
| go-zero | v1.4.4 | v1.7.x | ⚠️ 高 | 核心框架落后 6+ 版本 |
| gRPC | v1.53.0 | v1.60.x | ⚠️ 高 | 通信协议，API 变化 |
| GORM | v1.24.6 | v1.25.x | ⚠️ 中 | ORM 库，bug 修复 |
| go-redis | v8.11.5 | v8.x 兼容 | ⚠️ 中 | 配合 go-zero 升级 |
| etcd client | v3.5.5 | v3.5.latest | 📋 低 | 随框架升级 |

**约束条件：**
- 交易平台对稳定性要求极高
- 服务间强依赖，需协调升级
- 团队对 go-zero v1.4.x API 熟悉
- 升级期间需保持业务连续

**关键干系人：**
- 后端开发团队（执行升级）
- 测试团队（验证功能）
- 运维团队（部署变更）

## Goals / Non-Goals

**Goals:**
- 修复所有已知安全漏洞（特别是 golang.org/x/crypto）
- go-zero 升级到稳定版本（v1.7.x 或 v1.8.x），保持 API 兼容
- gRPC、GORM 等核心依赖适度升级，避免破坏性变更
- 全量回归测试通过，现有功能 100% 正常
- 建立依赖定期审计机制

**Non-Goals:**
- 不升级到 go-zero 最新版本（v1.10.x），避免过多破坏性变更
- 不重构业务代码或架构调整
- 不变更数据库 Schema 或数据迁移
- 不引入新的服务或模块
- 不升级基础设施版本（Docker Compose 中的服务保持现状）

## Decisions

### 1. 保守版本选择策略

**决策**: 选择"安全"的中间版本而非最新版

**理由:**
- go-zero v1.4.4 → v1.10.1 跨度太大，破坏性变更多
- v1.7.x - v1.8.x 是相对稳定的过渡版本，API 变化较小
- gRPC v1.60.x 是 v1.53.0 后的成熟版本，变更可控
- 后续可根据运行情况考虑是否进一步升级

**替代方案考虑:**
- 全部最新：风险过高，不推荐
- 仅安全补丁：go-zero 落后太多，适度升级获得必要改进

### 2. 升级顺序：安全优先，由底向上

**决策**:
```
1. golang.org/x/* (系统包，最底层)
2. go-zero (核心框架)
3. gRPC (依赖 go-zero)
4. GORM、go-redis (业务依赖)
5. etcd 及其他 (随框架依赖)
```

**理由:**
- 系统包安全漏洞优先级最高
- 框架升级影响最广，需充分测试
- 由底向上避免依赖冲突

### 3. 模块化升级策略

**决策**: 分模块逐个升级，非一次性全部更新

**步骤:**
1. 先升级 grpc-common、mscoin-common 公共模块
2. 升级非关键服务（如 jobcenter）验证
3. 升级核心服务（ucenter、market、exchange）
4. 最后升级 API 网关

**理由:**
- 公共模块无依赖，可独立升级
- jobcenter 是后台任务，影响面小
- 核心服务需充分验证
- API 网关直接对外，风险最高

### 4. 测试策略

**决策**: 三层测试保障

1. **编译验证**: `go build` 所有服务
2. **单元测试**: 运行现有单元测试
3. **集成测试**: 启动完整服务栈验证 gRPC 调用
4. **回归测试**: 核心业务场景手工验证

### 5. 回滚策略

**决策**: Git 标签 + Docker 镜像双重保障

- 升级前创建 Git 标签 `pre-upgrade-<date>`
- 成功升级后创建标签 `post-upgrade-<date>`
- 构建 Docker 镜像并标记版本
- 生产环境问题立即回滚至 `pre-upgrade` 标签

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| go-zero API 不兼容导致编译错误 | 高 | 预先阅读 changelog，准备代码修改预案 |
| gRPC 负载均衡行为变化 | 中 | 测试环境验证服务发现，监控调用延迟 |
| GORM SQL 生成变化影响性能 | 中 | 慢查询日志对比，性能基准测试 |
| 依赖版本冲突 | 中 | 使用 go mod tidy 清理，手动解决冲突 |
| 升级后服务不稳定 | 高 | 灰度发布，单节点先升级观察 |
| 回滚时间长 | 中 | 预构建回滚镜像，准备好回滚脚本 |
