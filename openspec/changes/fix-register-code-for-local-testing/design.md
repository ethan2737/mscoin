## Context

当前注册验证码由 `ucenter` 服务在发送验证码时生成并写入 Redis，注册提交时再从相同 key 读取后比对。现有本地测试依赖人工读取 Redis 或观察运行日志才能获得验证码，增加了联调和回归成本。此次变更仅针对本地测试可预测性，目标是在不改变现有注册接口和校验路径的前提下，将验证码生成值固定为 `123456`。

约束如下：
- 不修改 `/uc/mobile/code` 与 `/uc/register/phone` 的接口协议
- 不调整 Redis key、过期时间和注册校验顺序
- 不扩展为完整的环境开关或验证码体系重构

## Goals / Non-Goals

**Goals:**
- 让本地测试发送验证码后始终写入固定值 `123456`
- 保持注册提交仍通过 Redis 中的验证码进行校验，确保链路行为与真实流程一致
- 降低人工联调、冒烟验证和自动化测试的验证码获取成本

**Non-Goals:**
- 不改造短信发送平台接入方式
- 不修改前端表单、接口字段或 Redis 结构
- 不处理生产环境和测试环境切换策略

## Decisions

### 决策 1：仅替换验证码生成值，不调整发送与校验链路

选择方案：
在 `SendCode` 中直接写入固定验证码 `123456`，注册时继续读取 `REGISTER::<phone>` 并校验 `in.Code`。

备选方案：
- 在注册逻辑中绕过验证码校验
- 在前端写死验证码并跳过发送接口

理由：
- 改动最小，只影响验证码生成策略
- 保留真实 Redis 存取与注册校验链路，测试覆盖面更完整
- 避免前端与后端各自维护独立的测试特判

### 决策 2：保留现有 Redis key 与过期时间

选择方案：
继续使用 `REGISTER::<phone>` 作为缓存 key，过期时间保持 15 分钟。

备选方案：
- 使用单独的测试 key
- 取消过期时间

理由：
- 降低对现有注册逻辑和排障习惯的影响
- 便于复用已有 Redis 查询和 smoke check 方式
- 避免引入额外的兼容分支

## Risks / Trade-offs

- 本地固定验证码逻辑若被误带入非本地环境 → 后续实施时应明确限定使用场景并在文档中标注
- 测试人员可能误以为验证码仍是随机生成 → 需要在变更说明和验证步骤中明确固定值为 `123456`
- 旧进程复用可能让代码修改未生效 → 实施时需要配合重启 `ucenter` 服务验证实际运行版本
