## Context

当前交易页是 Vue 3 迁移后的页面，但只恢复了部分旧功能。运行态排查显示：

- K 线容器存在，但没有创建 TradingView widget，导致图表区域完全空白。
- 前端仍在请求 `/market/exchange-plate-mini`、`/market/exchange-plate-full`、`/market/latest-trade`，但 `market-api` 当前没有注册这些路由，浏览器实际返回 404。
- 深度图组件本身可用，但需要依赖完整盘口数据触发 `draw()`，否则只能显示空画布。
- 委托区调用链存在，且 `exchange-api` 路由和鉴权已接通，但本地联调数据与页面反馈不足，当前无法稳定证明“当前委托/历史委托/撤单”链路可用。
- 页面样式混用了 Vue 3 + Element Plus 组件和历史 `exchange.css` 中的 iView 主题规则，导致深色容器中出现白底表格和输入框，视觉与布局都不稳定。

这个改动跨前端页面、前端运行时、行情 API、联调基线和浏览器验收流程，属于典型的跨模块修复。

## Goals / Non-Goals

**Goals:**
- 恢复交易页 K 线图的初始化、历史数据加载和币对切换刷新。
- 在 `market-api` 暴露交易页依赖的盘口、深度和最新成交接口，并返回前端当前结构可直接消费的数据。
- 保证交易页在本地联调环境下，盘口、深度、最新成交和委托区具备可验证反馈。
- 统一交易页深色主题样式，消除白底组件混入和主要布局遮挡。
- 为交易页增加可重复的 smoke/QA 验证路径，避免再次出现“首页有行情、交易页半失效”的断层。

**Non-Goals:**
- 不在本次变更中重做整站设计系统。
- 不改造撮合引擎或新增复杂下单能力。
- 不追求交易页所有历史功能完全回归，只覆盖当前主链路必须可用的部分。
- 不把现货交易页和合约交易页一起重构。

## Decisions

### 1. 沿用现有交易页 API 语义，补齐 `market-api` 路由而不是直接改前端去吃新接口

交易页当前已经按 `/market/exchange-plate-mini`、`/market/exchange-plate-full`、`/market/latest-trade` 组织数据消费。相比直接把前端整体改造成另一套接口，这次优先在 `market-api` 恢复兼容路由，让页面先回到可运行状态。

这样做的原因：
- 交易页已有完整的数据映射和 websocket 刷新逻辑，修改成本更低。
- 可以减少对页面模板和交互层的扰动，把风险集中在 API 补齐和图表恢复上。
- 后续如果要统一到另一套行情 API，再单独做协议收敛。

备选方案：
- 直接让交易页改成调用 `/market/symbol-thumb-trend` 等现有接口并重写映射。缺点是改动面更大，而且不能解决盘口和最新成交接口缺失的问题。

### 2. 以旧 Vue2 页面为行为参考，恢复 Vue3 页面的 TradingView 初始化

旧页面已经包含稳定的 TradingView 配置、分时/周期切换和 datafeed 接法。Vue3 页面当前缺失的是“实例创建”这一步，不是整体交互设计缺失。因此本次实现应对照旧页面，把 K 线初始化最小必要逻辑迁回 Vue3 页面。

这样做的原因：
- 旧页面已经证明这套图表接法在本项目里是跑通过的。
- 能最大限度复用现有 `bitrade.js` datafeed 和静态资源目录。
- 避免在本次修复里引入新的图表库或重写数据协议。

备选方案：
- 更换为其他图表组件。缺点是范围失控，且会引入新的外部依赖和更大测试面。

### 3. 深度图继续使用现有 `DepthGraph.vue`，但改为由完整盘口接口驱动

深度图组件当前不是主要问题，问题在于它拿不到真实 `ask/bid.items`。因此实现上应保持组件不变，通过 `getPlateFull()` 和 websocket 刷新喂给它完整盘口数据。

这样做的原因：
- 风险最小，问题边界清晰。
- 可以让盘口表格和深度图共享同一套行情来源，减少口径偏差。

备选方案：
- 重写深度图组件。缺点是收益小，不针对当前根因。

### 4. 委托区按“链路可验证”定义完成，而不是强行伪造交易行为

委托区问题要分两层处理：
- 页面必须正确调用当前委托、历史委托和撤单接口，并对登录态和空态给出明确反馈。
- 本地联调环境需要提供最小可验证数据，要么通过基线订单，要么通过 smoke 流程自动创建一笔委托，再读取列表验证。

这样做的原因：
- 当前“空列表”不一定是 bug，也可能只是没有数据基线。
- 把“接口正确”和“环境可验证”拆开，能避免以后再把联调空态误判成实现缺陷。

备选方案：
- 单纯在页面里显示 mock 行。缺点是会掩盖真实链路是否可用。

### 5. 交易页主题样式按 Element Plus 组件结构重新覆写，不继续依赖 iView 规则兜底

当前 `exchange.css` 主要覆盖 `.ivu-*`，对 `<el-table>`、`<el-input>` 等组件几乎无效。继续叠加旧样式只会让问题越来越隐蔽。本次应在 Vue3 交易页内明确覆写 Element Plus 表格、输入框、tabs、按钮、空态和布局容器的深色主题。

这样做的原因：
- 这是导致“深色背景 + 白色组件混杂”的直接原因。
- 局部收敛到交易页内部，比全局主题改造更安全。

备选方案：
- 引入全局 Element Plus 主题包。缺点是会影响全站其他页，不适合这次定点修复。

## Risks / Trade-offs

- [兼容路由恢复后与未来统一行情 API 重复] → 先以交易页恢复为目标，接口命名保持兼容；后续如需协议收敛，再单独提 change 清理冗余路由。
- [旧版 TradingView 逻辑迁回 Vue3 时引入额外运行时依赖] → 只迁最小必要逻辑，并在浏览器侧验证 `window.TradingView`、容器挂载和历史数据加载。
- [委托区基线如果直接写死订单，可能污染本地调试环境] → 优先使用可重复创建/清理的最小联调数据，不长期保留大量脏订单。
- [局部样式修复可能和旧 `exchange.css` 冲突] → 明确收口到 Vue3 交易页组件作用域和必要的 `:deep(...)` 覆写，避免继续堆叠两套主题规则。
- [只修主链路后，次要交互仍可能存在边角问题] → 在验收中把 K 线、盘口、深度、最新成交、委托和主要布局列为必须项，次要视觉细节后续再细化。
