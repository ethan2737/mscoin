## Context

当前 MSCOIN 前端采用 Vue 3 + Element Plus 技术栈，登录页已完成表单样式基线升级，使用统一的暗色输入容器样式、聚焦/错误态处理和 prepend 组合控件一体化设计。

注册页（`Register.vue`）仍使用旧版样式写法，仅简单覆盖 Element Plus 默认样式为透明背景，但未补充完整的暗色主题样式定义，导致：
- 输入框背景色不一致（部分白底、部分透明）
- 缺少 hover/focus/error 态视觉反馈
- 区号选择框与输入框视觉分离
- 验证码按钮风格突兀
- 页面布局未垂直居中

**约束条件：**
- 不得修改登录页已验证可用的样式基线
- 保持现有注册业务逻辑和 API 调用不变
- 仅调整样式，不引入新依赖

## Goals / Non-Goals

**Goals:**
- 注册页表单样式与登录页视觉基线对齐
- 输入框/选择框统一的暗色背景、描边、状态反馈
- prepend 组合控件（区号、邀请码）视觉一体化
- 验证码按钮改为暗色次级按钮风格
- 页面改为垂直居中卡片布局

**Non-Goals:**
- 不修改注册 API 接口或验证逻辑
- 不调整表单字段结构或校验规则
- 不引入新的 UI 组件库或主题系统
- 不涉及移动端响应式布局调整

## Decisions

### 决策 1：直接复用登录页样式基线而非重新设计

**选择方案：** 将登录页已验证的 `.auth-form-item` 样式类迁移到注册页，为每个 `el-form-item` 添加对应 class。

**替代方案：** 为注册页单独创建一套 `register-form-item` 样式。

**理由：**
- 登录页样式已在线上环境验证可用
- 减少重复代码，便于后续统一维护
- 避免两套表单样式继续分化

### 决策 2：保持现有 DOM 结构不变

**选择方案：** 仅修改 `<style>` 部分，通过 CSS 选择器精准覆盖目标元素样式。

**替代方案：** 重构注册页 DOM 结构以匹配登录页。

**理由：**
- 降低回归风险
- 减少代码审查复杂度
- DOM 结构差异不影响样式复用

### 决策 3：验证码按钮采用暗色描边按钮而非填充按钮

**选择方案：** 按钮背景色使用 `#27313e`，文字颜色 `#fff`，hover 时改为品牌色 `#f0a70a`。

**替代方案：** 使用与登录按钮一致的品牌色填充按钮。

**理由：**
- 次级操作（发送验证码）视觉权重应低于主操作（注册）
- 与登录页验证码按钮风格保持一致
- 减少视觉干扰，突出注册主按钮

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 注册页原有自定义样式可能与新基线冲突 | 逐字段视觉回归验证，必要时调整选择器优先级 |
| Element Plus 组件升级可能导致样式覆盖失效 | 添加样式回归测试，监控视觉变化 |
| 暗色主题可能与部分浏览器默认样式冲突 | 在主流浏览器（Chrome/Edge/Firefox）进行跨浏览器验证 |
| 移动端布局可能受居中卡片影响 | 限制居中布局仅适用于桌面端，移动端保持原有流式布局 |
